Question 1: semantic HTML and accessibility

What is the best first accessibility choice for a Frontend Engineer? Consider this during local development. Difficulty focus: beginner.

Answer choices

  • Use ARIA roles on every div.
  • Remove focus styles for a cleaner visual design.
  • Use only color to show validation errors.
  • Use native semantic elements before adding custom ARIA.

Correct answer: Use native semantic elements before adding custom ARIA.

Explanation: Native elements bring keyboard behavior, semantics, and browser support by default. This is why the scenario matters for Frontend Engineer work.

Question 2: React component design

What makes a React component easier to reuse? Consider this during code review. Difficulty focus: beginner.

Answer choices

  • Hardcoding copy and API URLs.
  • Putting all page logic into one component.
  • Clear props, limited responsibility, and predictable rendering states.
  • Reading every value from global state.

Correct answer: Clear props, limited responsibility, and predictable rendering states.

Explanation: Reusable components have clean boundaries and do not know too much about their parent page. This is why the scenario matters for Frontend Engineer work.

Question 3: state and forms

What should a robust form handle besides happy-path typing? Consider this after a failed deployment. Difficulty focus: beginner.

Answer choices

  • Only the final API response.
  • Validation, loading, errors, empty states, and draft persistence when useful.
  • Only the submit button style.
  • Only desktop screen sizes.

Correct answer: Validation, loading, errors, empty states, and draft persistence when useful.

Explanation: Real users pause, make mistakes, lose network, and return later. This is why the scenario matters for Frontend Engineer work.

Question 4: API integration

What is a production-ready data fetching state? Consider this while debugging production-like data. Difficulty focus: beginner.

Answer choices

  • It handles loading, success, empty, error, retry, and stale data behavior.
  • It assumes the network always succeeds.
  • It blocks the entire app for one request.
  • It logs errors only to the console.

Correct answer: It handles loading, success, empty, error, retry, and stale data behavior.

Explanation: Data fetching is part of the user experience, not only a technical detail. This is why the scenario matters for Frontend Engineer work.

Question 5: performance

What should you do before performance optimization? Consider this when explaining the work to a hiring manager. Difficulty focus: beginner.

Answer choices

  • Rewrite the app in a new framework immediately.
  • Memoize every component.
  • Remove all animations from every page.
  • Measure the bottleneck with real tools and realistic user conditions.

Correct answer: Measure the bottleneck with real tools and realistic user conditions.

Explanation: Measurement prevents time spent optimizing the wrong thing. This is why the scenario matters for Frontend Engineer work.

Question 6: responsive product design

What is the best responsive layout habit? Consider this while reducing operational risk. Difficulty focus: beginner.

Answer choices

  • Hide important actions on mobile.
  • Use fixed pixel widths everywhere.
  • Design stable layout rules for common content lengths across mobile and desktop.
  • Scale all font sizes with viewport width.

Correct answer: Design stable layout rules for common content lengths across mobile and desktop.

Explanation: Responsive UI needs planned constraints, not accidental shrinking. This is why the scenario matters for Frontend Engineer work.
